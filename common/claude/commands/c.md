## 전달된 인자

`$ARGUMENTS`

> 인자에 `pr`, `PR`, `--pr` 같은 키워드가 들어 있으면 **PR 모드**다. 커밋 완료 후 5단계(PR 생성)까지 진행한다. 그 외에는 커밋만 한다.

**현재 브랜치:**
!`git branch --show-current 2>/dev/null || echo "detached HEAD or no branch"`

**Git Status:**
!`git status --short`

**Staged Changes:**
!`git diff --cached --stat`

**Unstaged Changes:**
!`git diff --stat`

**전체 Diff (staged + unstaged):**
!`git diff HEAD`

**최근 커밋 스타일 참고:**
!`git log --oneline -5 2>/dev/null || echo "No commits yet"`

**로컬 브랜치 목록:**
!`git branch 2>/dev/null | head -20 || echo "no branches"`

**원격 저장소:**
!`git remote -v 2>/dev/null | head -4 || echo "no remote"`

!`gh auth status 2>&1 | head -5 || echo "gh not available"`

## 작업 지침

### 핵심 원칙: 하나의 커밋 = 하나의 의도, 하나의 브랜치 = 하나의 컨텍스트

**커밋 분리가 기본이다.** 여러 의도가 섞여 있으면 반드시 커밋을 분리한다.
**브랜치 분리는 보수적이다.** 컨텍스트(맥락/도메인/머지 타이밍)가 명확히 다를 때만 브랜치까지 분리한다.
**PR 생성은 옵트인이다.** 인자에 `pr`/`PR`/`--pr`이 있을 때만 5단계까지 진행, 없으면 4단계에서 종료.

전체 흐름:
- 1단계: 변경사항을 커밋 단위로 분리
- 1.5단계: 커밋들의 타겟 브랜치 결정 (같은 브랜치 vs. 별도 브랜치)
- 2단계: 각 커밋의 메시지 + 타겟 브랜치 제안 (PR 모드면 PR 제목/본문 초안도 함께)
- 3단계: 유저 컨펌 (PR 모드면 PR 내용까지 함께 컨펌)
- 4단계: 커밋 실행
- 5단계: (PR 모드일 때만) push + PR 생성

**PR 모드 특수 케이스 — 워킹트리 클린:**
- 변경사항이 하나도 없으면 1~4단계는 건너뛰고, 현재 브랜치의 미푸시 커밋을 기반으로 5단계만 실행한다.
- 미푸시 커밋도 없다면 사용자에게 "커밋도 없고 push할 것도 없다"고 알리고 종료.

### 1단계: Diff 분석 및 커밋 단위 분리

위의 변경사항을 분석하여 **서로 다른 의도/목적**을 가진 변경들을 식별해라.

**반드시 분리해야 하는 경우:**
- 서로 다른 기능을 추가/수정한 경우 (예: A 기능 추가 + B 기능 수정 → 2개 커밋)
- 버그 수정과 기능 추가가 섞인 경우
- 리팩토링과 기능 변경이 섞인 경우
- 설정 변경과 코드 변경이 섞인 경우
- 서로 다른 모듈/컴포넌트를 독립적으로 수정한 경우
- 여러 개의 독립적인 버그를 수정한 경우

**단일 커밋이 허용되는 경우 (엄격하게 적용):**
- 변경 파일이 1-2개이고, 하나의 명확한 목적만 있을 때
- 하나의 기능을 위해 여러 파일을 함께 수정해야 할 때 (예: 컴포넌트 + 스타일 + 테스트)
- 단순 오타 수정, 포맷팅 등 사소한 변경일 때

**의도(intent) 분류 예시 + 선택 규칙:**
- `feat`: 새 기능 추가
- `fix`: 버그 수정
- `refactor`: 동작 변경 없는 코드 개선
- `style`: 포맷팅, 세미콜론 등
- `docs`: 문서 변경
- `chore`: 설정, 빌드 등
- `test`: 테스트 추가/수정

**Breaking Change 표시 (필수):**
하위 호환성을 깨는 변경이 있으면 반드시 표시해라:
- API 시그니처 변경 (파라미터 추가/제거/타입 변경)
- 기존 기능 제거 또는 동작 방식 변경
- 설정/환경 변수 형식 변경
- 필수 의존성 변경

표시 방법 (둘 중 하나 선택):
- 제목 타입 뒤에 느낌표 추가: `feat!: remove deprecated login API`
- 본문에 `BREAKING CHANGE:` 추가:
   ```
   BREAKING CHANGE: login() 함수의 파라미터가 변경됨
   ```

→ **의도가 다르면 무조건 분리!**

feat vs fix 판단 체크리스트 (헷갈릴 때 이 순서로 결정):
- 기존 동작이 깨졌거나 잘못된 로직을 바로잡는가? → `fix`
- 새로운 기능/옵션/플로우가 추가되어 사용 범위가 넓어지는가? → `feat`
- 동작은 같고 구조만 개선/정리했는가? → `refactor`
- 설정/CI/빌드/의존성만 건드렸는가? → `chore`
- 코드 없이 문서만 수정했는가? → `docs`
- 테스트 코드만 추가/수정이고 제품 동작은 그대로인가? → `test`
- 위에 명확히 안 맞으면 AskUserQuestion으로 용도(버그 수정인지, 기능 추가인지) 확인 후 타입 결정. 모호할 때 `feat` 대신 `fix`/`refactor`/`chore` 중 해당하는 것을 우선 검토.

### 1.5단계: 브랜치 분리 판단

식별된 커밋들을 보고 **다른 브랜치로 분리할지** 판단한다.

**기본은 같은 브랜치다.** 다음 중 하나라도 명확히 해당될 때만 분리 제안:

- 완전히 독립적인 도메인의 작업 (예: 인증 시스템 개선 + 결제 UI 리디자인)
- PR을 따로 올리는 게 자연스러운 작업 (다른 리뷰어, 다른 머지 타이밍)
- 핫픽스와 일반 기능 작업이 섞임 (핫픽스는 빨리 머지되어야 함)
- 우연히 한 번에 작업했지만 서로 코드 의존성이 없음

**같은 브랜치를 유지하는 경우 (기본값):**
- 같은 기능/리팩토링의 일부 (예: A 기능 코드 + A 관련 테스트 + A 문서)
- 순차적으로 의존성이 있는 작업 (B가 A의 코드를 사용)
- 작은 fix들이 함께 머지되어도 무방한 경우
- 단일 커밋만 있는 경우
- 같은 파일에 여러 의도가 섞여 있어 파일 단위로 깨끗하게 나뉘지 않는 경우

**판단 체크리스트 (헷갈릴 때 이 순서로 결정):**
- 두 작업을 별도 PR로 올릴 가능성이 매우 높은가? → 분리 제안
- 두 작업의 머지 우선순위/타이밍이 다른가? → 분리 제안
- 한 작업이 다른 작업의 코드/타입에 의존하는가? → 같은 브랜치
- 같은 파일에 양쪽 변경이 섞여 있는가? → 같은 브랜치 (파일 단위 분리 불가)
- 헷갈리면 → 같은 브랜치 (덜 침습적; 사용자가 나중에 더 쉽게 분리 가능)

**파일 충돌 사전 체크 (필수):**
- 서로 다른 타겟 브랜치로 갈 커밋들이 **같은 파일**을 건드린다면, 그 분리는 안전하지 않다.
- 이 경우 자동으로 같은 브랜치에 묶고, 사용자에게 "파일 겹침 때문에 같은 브랜치로 처리한다"고 알린다.

**브랜치 이름 규칙 (제안 시):**
- 컨벤셔널 prefix 사용: `feat/`, `fix/`, `refactor/`, `chore/`, `docs/` 등
- 케밥케이스, 짧고 명확하게
- 예: `feat/user-auth`, `fix/login-redirect`, `refactor/payment-module`
- 현재 브랜치가 그 자체로 의미 있는 작업 브랜치(`feat/...`, `fix/...`)이면 그 브랜치를 유지하고, 분리되어 나가는 쪽이 새 브랜치를 가져가는 게 자연스럽다.
- 현재 브랜치가 `main`/`master`/`develop` 같은 공유 브랜치라면, **모든 그룹**을 새 브랜치로 빼는 것을 고려한다 (공유 브랜치에 직접 커밋하지 않는 게 일반적 워크플로).

### 2단계: 커밋 메시지 제안

각 커밋별로 다음 형식으로 제안해:

**커밋 N:**
- **타겟 브랜치**: `<현재 브랜치명>` (그대로) 또는 `<새 브랜치명>` (현재 브랜치에서 분기)
- 포함할 파일들: `file1.ts`, `file2.ts`
- 의도: (feat/fix/refactor 등)
- Breaking Change 여부: 있음/없음 (있으면 무엇이 깨지는지 명시)
- 제목 (영어, 50자 이내): `feat: add user authentication flow` (breaking change면 `feat!:` 사용)
- 본문 (한글, 2 depth bullet list):
  - 주요 변경사항
    - 세부 내용 1
    - 세부 내용 2
  - (breaking change가 있으면) BREAKING CHANGE: 설명

(모든 커밋에 대해 반복)

**브랜치 분리가 있는 경우, 브랜치 단위로 한 번 더 요약:**

```
브랜치 `<현재 브랜치명>` (그대로):
  - 커밋 1: <제목>
  - 커밋 2: <제목>

브랜치 `<새 브랜치명>` (`<현재 브랜치명>`에서 분기):
  - 커밋 3: <제목>
```

**단일 커밋인 경우에도 위 형식을 따르되, "커밋 1"만 제안하면 됨.**

메시지 제안 시 타입 자가 검증:
- 제안한 타입이 체크리스트와 일치하는지 다시 확인하고, 기능 추가가 아닌 단순 버그 수정이면 `feat`가 아닌 `fix`로 수정.
- 변경 요약에 테스트/문서 반영 여부를 함께 적어 빠진 부분이 없는지 점검.

**PR 모드 추가 작업 — 각 PR 단위(= 브랜치)마다 PR 제목/본문 초안도 함께 제안:**

PR 단위는 "main/master/develop이 아닌 각 타겟 브랜치"다. 같은 브랜치에 여러 커밋이 묶이면 그 커밋들을 모아 하나의 PR로 만든다.

PR 제목 (영어, 짧고 명확하게, ~70자 이내):
- 단일 커밋 브랜치: 그 커밋의 제목을 그대로 사용
- 여러 커밋 브랜치: 가장 핵심적인 변경을 요약해 새로 작성 (커밋 타입 prefix 동일 컨벤션)

PR 본문 (**한글 위주**, 영어는 식별자/기술 용어/외래어처럼 한글로 옮기면 부자연스러운 부분에서만 사용):

```markdown
## 변경 요약
<무엇이 바뀌었는지 1~2줄로 응축>

## 무엇이 / 어떻게 바뀌었나
- <변경 포인트 1>
  - <세부 구현/파일/함수>
- <변경 포인트 2>
  - <세부 구현/파일/함수>

## 왜 바뀌었나
<배경, 동기, 해결하려는 문제. 이슈/티켓 번호가 있으면 같이>

## 주의할 점
- <호환성 영향, breaking change, 마이그레이션, 알려진 한계, 후속 작업 등>
- <리뷰어가 특히 봐줬으면 하는 부분이 있으면 명시>

## 테스트 / 검증
- [ ] <확인 항목 1>
- [ ] <확인 항목 2>
```

**PR 본문 작성 원칙:**
- "무엇이 / 어떻게 / 왜 / 주의할 점" 4개 축을 항상 채워라. 비어 있으면 그 섹션을 빼는 게 아니라, 비어 있어도 되는지 한 번 더 생각해라.
- 코드를 다시 읽지 않아도 리뷰어가 변경 의도를 이해할 수 있을 정도의 정보를 담아라.
- 영어 코드 식별자(함수명, 변수명, 파일 경로 등)는 그대로 쓰고, 설명은 한글로.
- 외부 의존성, 환경 변수, 마이그레이션 필요 사항은 "주의할 점"에 빠짐없이 적어라.
- 사소한 변경(오타, 포맷팅 등)은 본문도 짧게 — 형식을 위한 빈칸 채우기 금지.

### 3단계: 유저 컨펌

AskUserQuestion 도구를 사용해서 유저에게 다음을 물어봐:
- 제안한 커밋 분리 / 브랜치 분리 방식으로 진행할지
- 커밋 메시지 수정이 필요한지
- 커밋을 더 합치거나 나눌지
- 브랜치 분리/병합 여부 변경 (분리한 걸 같은 브랜치로 합치기, 또는 같은 브랜치를 분리하기)
- 새 브랜치 이름 수정
- 취소할지

**PR 모드 추가 컨펌 항목:**
- PR 제목/본문 초안이 적절한지, 수정 필요 여부
- Draft PR로 올릴지 일반 PR로 올릴지 (기본: 일반)
- PR base 브랜치 (기본: origin/HEAD가 가리키는 default branch, gh가 자동 결정)
- 커밋만 하고 PR은 생략할지 (유저가 마음을 바꿀 수 있는 escape hatch)

### 4단계: 커밋 실행

**Case A — 단일 브랜치 (브랜치 분리 없음):**

각 커밋을 순차적으로 실행:
1. 해당 커밋에 포함할 파일만 staging (`git add <files>`)
2. 해당 커밋 메시지로 커밋
3. 다음 커밋으로 진행
4. 모든 커밋이 완료될 때까지 반복

**Case B — 브랜치 분리:**

전제 조건 (시작 전에 확인):
- 모든 변경사항이 unstaged 상태인 게 가장 안전 (staged인 게 있으면 사용자에게 알리고 진행 여부 확인).
- 1.5단계에서 파일 겹침이 없음을 이미 확인.
- 새 브랜치 이름이 이미 존재하지 않는지 `git show-ref --verify --quiet refs/heads/<name>`로 확인. 존재하면 사용자에게 알리고 다른 이름 받기.

원본 브랜치 기록:
```bash
ORIG_BRANCH=$(git branch --show-current)
```

**실행 순서:**

1. **원본 브랜치에 남기는 커밋부터 처리 (있다면):**
   - 원본 브랜치(`ORIG_BRANCH`)에 둘 커밋들을 순서대로 `git add <files>` + `git commit -m "..."`.
   - 이렇게 하면 새 브랜치를 만들 시점에 원본 브랜치에 staged 잔여물이 없는 깨끗한 상태가 된다.
   - working tree에는 아직 처리되지 않은 (다른 브랜치로 갈) unstaged 변경이 남아 있다.

2. **각 새 브랜치마다 순서대로:**

   a. 새 브랜치 생성: `git checkout -b <new-branch>`
      - 이 시점에 staged 변경이 없어야 한다 (1단계에서 정리됨).
      - working tree의 unstaged 변경은 그대로 따라온다.

   b. 해당 브랜치 커밋들을 순서대로 처리:
      - `git add <files for this commit>`
      - `git commit -m "..."`

   c. 원본 브랜치로 복귀: `git checkout <ORIG_BRANCH>`
      - 방금 새 브랜치에서 커밋한 파일들의 변경은 working tree에서 자동으로 원래 상태로 돌아간다 (커밋되었으므로 working tree에 unstaged로 남지 않음).
      - 아직 처리되지 않은 다른 브랜치들의 unstaged 변경은 계속 working tree에 남는다 — 다음 새 브랜치에서 사용됨.

3. **모든 새 브랜치 처리 완료 후:**
   - 원본 브랜치(`ORIG_BRANCH`)로 돌아와 있는지 확인.
   - `git status` 확인 — working tree가 의도한 대로 깨끗한지 검증.
   - 각 새 브랜치의 `git log --oneline -n 5` 요약을 보여주기.
   - 원본 브랜치의 `git log --oneline -n 5`도 보여주기.

**브랜치 분리 중 발생 가능 문제 및 대응:**

- `git checkout`이 conflict로 실패: 즉시 멈추고 사용자에게 현재 상태(`git status`)를 보여주며 결정 받기. 절대 자동으로 `--force`, `reset --hard`, `checkout -f` 같은 destructive 명령 사용 금지.
- 같은 파일이 여러 브랜치에 걸친 게 1.5단계에서 누락되어 실행 단계에서 발견된 경우: 멈추고 사용자에게 알리고 같은 브랜치로 묶는 옵션 제시.
- 새 브랜치 이름이 이미 존재: 멈추고 다른 이름 받기.
- 커밋 도중 hook 실패: 멈추고 사용자에게 원인 보여주기. 자동 `--no-verify` 사용 금지.

**커밋 메시지 형식 (단일/멀티 공통):**
```
제목

- 주요 변경사항
  - 세부 내용 1
  - 세부 내용 2
```

### 5단계: PR 생성 (PR 모드에서만)

PR 모드일 때, 4단계 커밋이 모두 성공한 뒤에만 진행한다. 4단계가 중간에 실패/중단되면 PR 생성도 건너뛴다.

**5.0 사전 검증 (실패 시 PR 생략하고 사용자에게 명확히 알린다):**
- `gh auth status`가 정상이고 인증되어 있는지 확인. 미인증이면 `gh auth login` 안내 후 종료.
- `git remote get-url origin`이 GitHub 호스트(github.com 또는 GHE)인지 확인. 아니면 PR 생략.
- PR 대상 브랜치가 1개 이상 식별되는지 확인. (대상 = main/master/develop이 아니고 origin 대비 ahead 커밋이 있는 브랜치)

**5.1 PR 대상 브랜치 산출:**
- 4단계에서 새로 만든 브랜치 + 원본 브랜치 중 main/master/develop이 아닌 것들이 대상.
- 워킹트리 클린 + PR 모드인 경우: 현재 브랜치만 대상 (origin/HEAD가 가리키는 base 대비 ahead 커밋이 있어야 함).
- 같은 브랜치에 묶인 여러 커밋은 **하나의 PR**로 생성한다.

**5.2 각 대상 브랜치마다 순차 실행:**

a. **브랜치로 이동 (이미 그 브랜치면 그대로):**
   ```bash
   git checkout <branch>
   ```

b. **Push:**
   - upstream이 없으면: `git push -u origin <branch>`
   - upstream이 있으면: `git push`
   - push 실패(non-fast-forward 등) 시 즉시 멈추고 사용자에게 `git status`/원인 보여주기. `--force` 자동 사용 금지.

c. **기존 PR 존재 여부 확인:**
   ```bash
   gh pr view <branch> --json url,state,number 2>/dev/null
   ```
   - 이미 열린 PR이 있으면: 새로 만들지 말고, 사용자에게 URL 보여준 뒤 "본문/제목을 업데이트할지" 묻기. 업데이트하면 `gh pr edit <number> --title "..." --body "..."`.
   - 없으면 다음 단계로.

d. **PR 생성:**
   ```bash
   gh pr create \
     --head <branch> \
     --title "<영어 PR 제목>" \
     --body "$(cat <<'EOF'
<한글 위주의 PR 본문 — 2단계에서 컨펌받은 내용>
EOF
)"
   ```
   - `--base`는 명시하지 않으면 gh가 default branch로 자동 설정 → 그대로 둔다. 사용자가 명시적으로 다른 base를 요구한 경우에만 `--base <name>` 추가.
   - Draft PR이면 `--draft` 추가.
   - 본문은 반드시 heredoc(`<<'EOF'`)으로 전달 — 따옴표/백틱/`$` 이스케이프 문제 회피.

e. **결과 확인:**
   - 명령 결과로 나오는 PR URL을 사용자에게 보여줄 수 있게 보관.

**5.3 전체 완료 후:**
- 생성/업데이트된 PR 목록(브랜치명, URL, 상태)을 표 형태로 정리해 보여준다.
- 원본 브랜치로 돌아와 있는지 확인 (`git branch --show-current`).

**PR 생성 중 발생 가능 문제 및 대응:**
- `gh` 미설치/미인증: PR만 생략하고 사용자에게 알린다. 커밋은 그대로 둔다.
- push 충돌: 멈추고 원인 보여주기. force push 자동 사용 금지.
- 동일 브랜치에 이미 PR이 있음: 5.2.c 절차대로 업데이트할지 컨펌 받기.
- gh API rate limit / 일시적 오류: 1회 자동 재시도까지만 허용, 이후엔 멈추고 사용자에게 알린다.
- base 브랜치가 명확하지 않음 (origin/HEAD가 비어 있음): 사용자에게 base를 물어본다.

**중요:**
- 유저가 취소하면 아무 커밋도 만들지 마.
- 유저가 메시지 수정을 원하면 수정된 내용으로 커밋해.
- 여러 커밋/브랜치 진행 중 문제가 생기면 즉시 멈추고 유저에게 현재 git 상태와 함께 알려.
- 자동 복구/롤백/force 동작은 하지 마. 모호한 상황에서는 사용자 판단을 받아라.
- 모든 작업 완료 후 결과(브랜치별 git log, PR 모드라면 PR URL까지)를 보여줘.
- **PR 모드여도 커밋은 PR과 독립적이다.** PR 생성이 실패해도 커밋은 그대로 살아 있으니 안내만 명확히 한다.
